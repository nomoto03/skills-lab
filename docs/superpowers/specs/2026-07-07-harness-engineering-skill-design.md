# harness-engineering スキル 設計ドキュメント

日付: 2026-07-07
状態: ユーザーレビュー待ち

## 目的

任意のリポジトリに対して「調査 → 質問 → 診断 → ハーネス設計の提案 → 承認された項目の実装 → 検証 → 改善バックログ提案」を行う汎用スキルを作る。エージェント(Claude Code)が安定して働ける環境をリポジトリごとに構築することがゴール。

## 確定した要件(ユーザーとのQ&Aによる決定事項)

| 論点 | 決定 |
|---|---|
| 成果物範囲 | 提案だけでなく、承認後に実装・設置まで行う |
| 対象環境 | Claude Code 専用(CLAUDE.md / .claude/settings.json / hooks / skills / subagents) |
| カバー領域 | 4領域すべて: コンテキスト整備 / 品質フィードバックループ / 権限・安全性 / 長時間実行・状態管理 |
| 配置 | まず skills-lab で開発・動作確認。配置先(~/.claude/skills 等)は後で決める |
| 対話量 | 要所だけ質問(コードから読めないことのみ、3〜5問) |
| 内部構造 | サブエージェント型(調査・設計を委譲、対話と実装はメインスレッド) |
| 追加要件 | 実装完了後に「ハーネスの効果を上げるためのリポジトリ側の改善バックログ」を提案する(Phase 7)。別スキルにはしない |

## 設計原則(Web調査から採用)

- **最小構成から始め、必要時のみ複雑化する**(Anthropic公式)。提案のデフォルトは最小実行可能ハーネス。
- **プロンプトではなく仕組みで強制する**。品質はフック・permissions・テストで担保し、CLAUDE.md は助言に徹する。
- **3層責任分離**: Intent層(CLAUDE.md=助言) / Harness層(permissions・hooks=実行前ゲート) / Environment層(sandbox=最終防御線)。
- **CLAUDE.md は50行以下・ポインタ形式**。「この行を消すとエージェントが失敗するか?」の削除テストに耐える行のみ残す。
- **成功宣言を自己申告にさせない**: default-FAIL の機能リスト、独立evaluator、Stopフックでのテスト強制。
- **フィードバック速度の階層**: PostToolUseフック(ms) → pre-commit(秒) → CI(分) → 人間レビュー(時間)。速い層に検査を寄せる。

## アーキテクチャ: 7フェーズ

```
Phase 1 調査     Exploreサブエージェント(read-only)に委譲 → harness-survey.md
Phase 2 質問     メインスレッドで AskUserQuestion(3〜5問、調査で分かることは聞かない)
Phase 3 診断+設計 architectサブエージェントに委譲 → harness-proposal.md
Phase 4 承認     メインスレッドで提案を提示、multiSelectで採用項目を選択
Phase 5 実装     メインスレッドで承認項目のみ templates/ ベースで適用
Phase 6 検証     hooks発火確認・settings.jsonパース確認・CLAUDE.md行数確認 → 最終レポート
Phase 7 バックログ ハーネスの効果を上げるためのリポジトリ側の宿題を優先度付きで提案(実装はしない)
```

- 質問(2)と承認(4)はサブエージェントが対話できないため必ずメインスレッドで行う。
- 中間ファイル(harness-survey.md / harness-proposal.md)はセッションのスクラッチパッドに置き、ファイルパスでサブエージェントに受け渡す。対象リポジトリを汚さない。
- 再実行に対応(冪等): 既存の .claude/ や CLAUDE.md がある場合は「新規構築」ではなく「差分診断」モードで動き、前回からの改善を検出してルーブリック再評価する。

### Phase 1: 調査(Explore サブエージェント)

調査項目:
- 言語 / フレームワーク / パッケージマネージャ、ビルド・テスト・リンタ・型チェックの各コマンドと所要時間(実際に実行して確認)
- 既存の .claude/(settings.json, hooks, skills, agents)と CLAUDE.md / AGENTS.md の有無・行数・鮮度
- CI設定、pre-commitフック、テストの実態(存在するか、通るか、何秒かかるか)
- git管理の有無、モノレポ構造、生成コード、シークレット(.env等)の場所
- OS/シェル環境(Windows/PowerShell か POSIX か — hooksテンプレの選択に必要)

出力: harness-survey.md(構造化レポート)

### Phase 2: 質問(メインスレッド)

references/interview.md の質問バンクから、調査結果に応じて3〜5問を選択:
1. 用途: 任せたい作業(バグ修正/機能開発/リファクタ/長時間自律実行)
2. 自律度: 全承認 / 安全な操作は自動 / サンドボックスで完全自動
3. 痛みポイント: 指示の繰り返し / 虚偽の完了報告 / 危険操作への不安 / 長作業での文脈喪失
4. 共有範囲: 個人用か、チームでgit共有か
5. 投資意思(テスト無しリポジトリのみ): テスト整備から始める意思があるか

ルール: 調査で判明したことは聞かない。回答はPhase 3への入力とする。

### Phase 3: 診断+設計(architect サブエージェント)

入力: harness-survey.md + ユーザー回答 + references/rubric.md + references/knowledge.md
出力: harness-proposal.md。要件:
- ルーブリック全項目に「現状OK / 要改善 / 対象外」の判定と根拠を付ける
- 改善提案は 効果(高/中/低) × 導入コスト(高/中/低) で優先度付け
- 各提案に「これを入れないと何が起きるか」を1行で明記
- 用途に合わない領域(例: バグ修正中心なのに長時間実行系)は「対象外」として理由とともに除外

### Phase 4: 承認(メインスレッド)

提案書を領域ごとに提示し、AskUserQuestion(multiSelect)で採用項目を選択させる。全採用・全却下も可能。

### Phase 5: 実装(メインスレッド)

- 承認された項目のみ実装。templates/ を出発点に、Phase 1 で確認済みの実コマンド・実パスに必ず書き換える(テンプレの丸コピー禁止)
- 既存ファイル(CLAUDE.md等)の変更は差分を提示してから適用
- 対象リポジトリがgit管理下なら、適用前の状態に戻せることを確認(コミットはユーザーの指示がある場合のみ)

### Phase 6: 検証

- hooks: ダミー操作で実際に発火するか確認(発火しない場合は実行権限・パスを疑う)
- settings.json: JSONとして妥当か、claude起動でエラーが出ないか
- CLAUDE.md: 50行以下か、各行が削除テストに耐えるか
- 結果を最終レポートとして提示。検証できなかった項目は「未検証」と明記する(成功と偽らない)

### Phase 7: 改善バックログ提案(実装はしない)

診断・実装の過程で見つかった「ハーネスの前提となるリポジトリ側の不足」を優先度付きバックログとして提示する。例:
- テストが無い/遅い → 品質ループの完了強制が組めない
- リンタが遅い(ESLint等) → 高速リンタ(oxlint/ruff等)への移行でPostToolUse化が可能になる
- 腐った説明文書 → 削除してテスト/型/ADRへの置き換え
- CIが無い → Stopフックの次の防衛線が欠けている

各項目に「これを解消すると次回このスキルを再実行したとき何が可能になるか」を付記する。実装は通常の開発作業として別途行う(このスキルのスコープ外)。

## スキルのファイル構成

```
harness-engineering/
├── SKILL.md                        # ワークフロー本体(7フェーズのオーケストレーション)
├── references/
│   ├── knowledge.md                # ハーネス設計の知見集(本設計の「設計原則」を詳細化)
│   ├── rubric.md                   # 4領域 × 5〜8項目の診断ルーブリック(PASS基準付き)
│   └── interview.md                # 質問バンク(選択条件・選択肢付き)
└── templates/
    ├── claude-md-skeleton.md       # 50行以下・ポインタ形式の雛形
    ├── settings-permissions.json   # deny/ask/allow 3段階プリセット(承認優先/許可リスト/サンドボックス)
    ├── hooks/
    │   ├── post-edit-lint.sh       # PostToolUse: 編集後リンタ自動実行
    │   ├── protect-config.sh       # PreToolUse: リンタ設定・.env改変ブロック
    │   ├── commit-on-stop.sh       # Stop: セッション終了時の自動コミット
    │   └── (各 .ps1 版)            # Windows/PowerShell環境用
    ├── progress/
    │   ├── PROGRESS.md             # セッション間ハンドオフ用雛形
    │   └── feature-list.json       # default-FAIL 機能リスト雛形
    └── agents/
        └── evaluator.md            # 読み取り専用の独立評価サブエージェント定義
```

## 診断ルーブリック(骨子 — 実装時に各領域5〜8項目へ拡充)

| 領域 | 項目例 | PASS基準 |
|---|---|---|
| コンテキスト | CLAUDE.mdの品質 | 存在し50行以下、コマンド・規約・ポインタ中心、腐った記述なし |
| コンテキスト | 知識の置き場所 | 説明文書でなくテスト/型/ADRで表現 |
| 品質ループ | 編集時フィードバック | PostToolUseでリンタ/型チェックが数秒以内に自動実行 |
| 品質ループ | 完了の強制 | テスト合格がプロンプト依存でなくフック/CIで強制 |
| 権限・安全 | deny基盤 | 破壊的コマンド・シークレット読取がpermissions.denyにある |
| 権限・安全 | 設定の共有 | .claude/settings.json がgit管理されている |
| 長時間実行 | 状態引き継ぎ | PROGRESS.md+git履歴で新セッションが状況を再構築できる |
| 長時間実行 | 自己評価の分離 | 成功判定が独立評価器/テストにある |

## エラー処理・フォールバック

| 状況 | 対応 |
|---|---|
| 対象リポジトリがgit未管理 | 調査で検出し、git init をPhase 4で最優先提案(ハーネスの前提のため) |
| テストが存在しない | 品質ループ提案を縮退(リンタのみ)し、テスト整備をPhase 7バックログの筆頭へ |
| サブエージェントの調査失敗 | メインスレッドで縮退調査(主要ファイルのみ)にフォールバック |
| 既存CLAUDE.mdが巨大(150行超) | 全面書き換えは提案せず、削除テストに基づく縮小案を差分で提示 |
| Windows環境 | Phase 1でシェル環境を検出し、hooksは .ps1 テンプレを使用 |
| ユーザーが全提案を却下 | Phase 5-6をスキップし、Phase 7(バックログ)のみ提示して終了 |

## スキル自体の検証方法(skills-labでの動作確認)

1. 特性の異なるサンプルリポジトリを2つ用意(例: テスト有りのTypeScript Webアプリ / テスト無しのPythonスクリプト集)
2. スキルを実行し、各フェーズの成果物(survey / proposal / 実装物 / バックログ)を確認
3. チェック観点: 調査で分かることを質問していないか / 提案が最小構成か / テンプレが実コマンドに書き換わっているか / 検証が実際に行われたか
4. 同一リポジトリへの再実行で差分診断モードが機能するか

## スコープ外(明示)

- ハーネス以外のリポジトリ改善の実装(テスト整備・CI構築等) — Phase 7で提案のみ
- Claude Code以外のエージェント(Codex/Cursor等)対応
- Environment層の構築自体(Dockerサンドボックス等) — 推奨として提示するに留める
- スキルの配布・プラグイン化 — 配置先は開発完了後に別途決定
