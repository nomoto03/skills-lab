# loop-engineering スキル 設計ドキュメント

日付: 2026-07-07
状態: ユーザーレビュー待ち

## 目的

「このゴールをループで達成したい」と渡されたとき、ゴールを検証可能な終了条件に変換し、リポジトリの実行可能性を確認した上で、最適なループ方式を選定・設計・設置し、承認後に点火(実際に起動)して監視手順を引き渡す汎用スキルを作る。ハーネスエンジニアリングの一階上(ループ層)を担う。

## 確定した要件(ユーザーとのQ&Aによる決定事項)

| 論点 | 決定 |
|---|---|
| 基本形 | ゴール駆動型(リポジトリ整備型ではなく、タスク/ゴール単位でループを設計) |
| harness-engineeringとの関係 | 前提チェック内蔵(R1〜R7)。不足時はharness-engineeringの実行を提案するが依存はしない |
| 点火範囲 | 承認後に実際に起動まで行う。反復上限・予算・緊急停止の3点必須 |
| カバーする方式 | 4方式: セッション内(/goal, /loop) / Ralph型永続 / 外部スクリプト型 / スケジュール型 |
| 構成 | A案: メインスレッド中心。実行可能性調査のみExploreサブエージェントに軽量委譲 |
| Phase 6の初回観察 | 方式により省略可(下記Phase 6参照) |

## 設計原則(Web調査から採用)

- **ループの5要素**: ①検証可能な終了条件を持つ目標 ②環境と対話するツール ③コンテキスト管理(外部メモリ=状態ファイル) ④終了ロジック(成功/失敗/反復上限/無進捗検出) ⑤エラー分別(回復可能/致命的→エスカレーション)
- **メーカー・チェッカー分離**: 成功判定を実装エージェントの自己申告にさせない。決定論的検証(コマンド)か独立evaluatorを使う
- **安全装置なしに点火しない**: 反復上限・予算・緊急停止手段(kill-switch)は必須。completion promiseは正確な文字列一致しかできないため、max-iterationsを主安全装置とする(Ralph公式プラグインの教訓)
- **トークン経済とレビュー帯域**: ループの実コストは反復数×トークンと「人間がレビューできるdiff量」。設計書に見積りを明記する
- **無進捗検出**: 同じ失敗の空回りを防ぐ。進捗メトリクス(例: passes:true件数、テスト成功数)がN反復変化しなければ失敗終了

## アーキテクチャ: 6フェーズ

```
Phase 1 ゴール検証可能化  メインスレッド。ゴール→コマンドで真偽判定できる終了条件へ変換+質問3〜4問
Phase 2 実行可能性チェック Exploreサブエージェント(軽量)。R1〜R7判定
Phase 3 方式選定          メインスレッド。decision-matrixで1方式を推奨、却下案も理由付きで提示
Phase 4 設計書承認        メインスレッド。loop-spec.mdを記入して提示、AskUserQuestionで承認
Phase 5 設置              部品生成(プロンプト/状態ファイル/evaluator/kill-switch)。{{MARKER}}置換必須
Phase 6 点火+監視引き渡し 起動→方式別の初回観察→監視手順(見方・介入・停止)を渡して終了
```

### Phase 1: ゴールの検証可能化(メインスレッド)

- ゴールを「コマンド1つで真偽判定できる終了条件」に変換する。例: 「APIを完成させる」→「feature-list.json の全項目 passes:true かつ npm test が緑」
- 変換できないゴール(「良い感じにする」等)は理由を示して停止し、人間参加型ループ(承認ゲート付き)への縮退を提案する
- ユーザーへの質問(3〜4問、references/interview.md から): ①無人時間帯と監視可否 ②トークン予算・反復上限の希望 ③失敗時のエスカレーション先(課題ファイル/通知) ④並列実行(worktree)の要否

### Phase 2: 実行可能性チェック(Explore サブエージェント、軽量)

references/readiness-check.md の R1〜R7 を判定し、結果をメインスレッドが `<scratchpad>/loop-readiness.md` に保存する。

| ID | 項目 | FAIL時 |
|---|---|---|
| R1 | 終了条件がコマンドで機械検証可能 | **点火不可**。Phase 1に戻る |
| R2 | 検証手段(テスト等)が存在し1反復あたり数分以内 | harness-engineering提案 or 縮退(リンタのみ) |
| R3 | git管理+専用ブランチ/worktreeで隔離可能 | git init / worktree作成を先行 |
| R4 | 無人実行に見合うpermissions(deny基盤) | 設置に含める(harnessテンプレの流用を案内) |
| R5 | 状態外部化の場所(PROGRESS.md/feature-list) | templatesから設置 |
| R6 | トークン予算とレビュー帯域が見積もられている | Phase 1の回答から算出して明記 |
| R7 | 緊急停止手段(kill-switch/プロセス停止手順) | **点火不可**。kill-switch設置で解消 |

- R1・R7 は点火ブロッカー(解消されるまでPhase 6に進まない)
- 不足が多い場合は harness-engineering スキルの実行を提案(依存はしない — 最低限の部品はこのスキルのtemplatesで補える)
- サブエージェント失敗時はメインスレッドで縮退チェック(R1・R3・R7のみ)にフォールバック

### Phase 3: 方式選定(メインスレッド)

references/decision-matrix.md に基づき1方式を推奨。採らなかった方式にも1行ずつ理由を付す。

| タスク特性 | 推奨方式 | 理由 |
|---|---|---|
| 数十分〜数時間で終わる見込み、条件明確 | `/goal` | 部品追加ゼロ、条件チェックは別モデルが毎ターン実施 |
| 外部状態のポーリング(CI待ち、定期チェック) | `/loop` | 間隔駆動が自然 |
| 複数セッション級の大型構築、夜間無人 | Ralph型 | Stopフック再投入で文脈リセット+ファイル継承。max-iterations必須 |
| 検証の独立性を強制したい | 外部スクリプト型 | evaluatorゲートを通らないと次反復に進めない構造 |
| 定期メンテ(依存更新、レポート生成)、ローカル不要 | schedule(cloud) | cron的定期実行、マシン不要 |

### Phase 4: ループ設計書の承認(メインスレッド)

templates/loop-spec.md を記入して提示し、AskUserQuestion で承認を得る。**反復上限・予算・緊急停止手段の3点が埋まっていない設計書では点火しない。** 設計書の必須項目:

- 終了条件(成功): 検証コマンドと期待値
- 終了条件(失敗): 反復上限 / N反復連続の無進捗(進捗メトリクス定義付き)
- 方式と却下案(理由付き)
- 予算(トークン相当・想定時間)
- 検証の独立性(決定論的コマンド or 独立evaluator)
- エスカレーション(失敗終了時の課題記録先・通知)
- 緊急停止手段と監視手順

### Phase 5: 設置

- 方式に応じた部品を templates/ から生成: ループプロンプト、状態ファイル、evaluator接続、kill-switch
- `{{MARKER}}` は必ず実値に置換(残したままの設置を禁止)
- 既存ファイルの変更は差分提示後に適用
- 対象リポジトリ側の成果物はループ用ブランチ/worktreeに隔離(R3)

### Phase 6: 点火+監視引き渡し

承認済み設計書の通りに起動(/goal発行、Ralph起動、スクリプト実行、schedule登録)。初回観察は方式別:

| 方式 | 初回観察 |
|---|---|
| セッション内(/goal, /loop) | **必須**: 1反復目の完了まで観察し異常がないことを確認 |
| 外部スクリプト型 | **必須**: 1反復目(claude -p 1回+evaluatorゲート1回)の完了まで観察 |
| Ralph型 | **起動確認のみ**: ループが開始され状態ファイルに最初の書き込みがあることを確認(1反復の完走は待たない) |
| schedule型 | **省略可**: 登録内容の確認のみ。初回実行後の確認手順を監視手順に含めて引き渡す |

最後に監視手順(進捗の見方・介入方法・停止方法)をユーザーに渡して終了する。

## スキルのファイル構成

```
skills/loop-engineering/
├── SKILL.md                    # 6フェーズのオーケストレーション
├── references/
│   ├── loop-knowledge.md       # ループ5要素・パターン・失敗モード・トークン経済・出典
│   ├── decision-matrix.md      # 方式選定マトリクス(タスク特性→方式)
│   ├── readiness-check.md      # R1〜R7チェックリスト(点火ブロッカー明示)
│   └── interview.md            # 質問バンク(無人時間帯/予算/エスカレーション/並列)
└── templates/
    ├── loop-spec.md            # ループ設計書テンプレ(Phase 4の承認対象)
    ├── ralph-prompt.md         # Ralph型PROMPT.md雛形(段階的目標+completion promise+escape hatch)
    ├── external-loop.sh        # bash while + claude -p + evaluatorゲート雛形
    ├── kill-switch.sh / .ps1   # AGENT_STOPファイル検知で全ツール呼び出しをブロックするPreToolUseフック
    └── monitoring.md           # 監視手順雛形(watchコマンド・確認頻度・介入・停止方法)
```

## エラー処理・フォールバック

| 状況 | 対応 |
|---|---|
| ゴールが検証可能化できない | 点火せず停止。人間参加型ループ(承認ゲート)への縮退を提案 |
| R1またはR7がFAILのまま | 点火不可。解消手順を提示して終了 |
| readiness調査のサブエージェント失敗 | メインスレッドでR1・R3・R7のみの縮退チェック |
| テスト無しリポジトリ | R2縮退(リンタのみ)+harness-engineering提案。終了条件はリンタ+目視レビューに変更 |
| 点火直後に異常(1反復目でエラー連発) | 即停止(kill-switch)し、原因と修正案を提示。自動リトライしない |
| Windows環境 | kill-switchは.ps1版を使用。external-loop.shはGit Bash前提と明記 |
| ユーザーが設計書を却下 | 修正して再提示。方式ごと変える場合はPhase 3へ戻る |

## harness-engineering との整合

- R3/R5 は harness ルーブリック D1/D2 に対応。R4 は C1 に対応
- evaluator・PROGRESS.md・feature-list.json は harness-engineering の templates と同型のものを想定(このスキル単独でも設置できるが、既に harness 整備済みならそれを使う)
- 併用フロー: harness-engineering でリポジトリを整備 → loop-engineering でゴールごとにループを設計・点火

## スキル自体の検証方法(skills-labでの動作確認)

1. fixtures/sample-ts(テスト有り)で「add関数にsubtract/multiply/divideを追加し全テスト緑」のような小ゴールを与え、/goal方式の選定→設計書→点火→完走を確認
2. fixtures/sample-py(テスト無し)でR2縮退とharness-engineering提案が出ることを確認
3. 検証可能化できないゴール(「コードを良い感じにして」)で点火拒否+縮退提案が出ることを確認
4. チェック観点: R1/R7未解消で点火しないか / 設計書に上限・予算・停止手段が必ず埋まるか / {{MARKER}}残りがないか / 監視手順が引き渡されるか

## スコープ外(明示)

- ハーネス自体の整備(→ harness-engineering スキル)
- ループで実行するタスクの中身の品質(それはループ内のエージェントとハーネスの責務)
- マルチエージェント・オーケストレーション基盤の構築(worktree並列は「1ゴール=1ループ」の範囲でのみ扱う)
- クラウド課金・トークン実測の自動追跡(見積りと上限設定のみ)
